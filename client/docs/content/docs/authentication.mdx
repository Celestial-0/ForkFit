## Authentication Model

ForkFit uses **session-based authentication** with **Bearer tokens**. When a user signs up or signs in, the API creates a server-side session and returns an opaque access token.

### How It Works

1. The client sends credentials to `/auth/signup` or `/auth/signin`
2. The server creates a session record in the database with a hashed token
3. The server returns the raw token in the response as `access_token`
4. The client includes this token in the `Authorization` header for all subsequent requests

```
Authorization: Bearer ff_<uuid>_<uuid>_<uuid>
```

<Callout type="info" title="Token Format">
  Session tokens follow the format `ff_<uuid>_<uuid>_<uuid>` — three concatenated UUIDv4 values prefixed with `ff_`. Tokens are hashed with SHA-256 before storage; the server never stores raw tokens.
</Callout>

## Making Authenticated Requests

Include the `Authorization` header with every request to a protected endpoint:

<Tabs items={['cURL', 'JavaScript', 'Python']}>
  <Tab value="cURL">
    ```bash
    curl -X GET http://localhost:3001/api/v1/auth/me \
      -H "Authorization: Bearer ff_abc123_def456_ghi789"
    ```
  </Tab>
  <Tab value="JavaScript">
    ```js
    const response = await fetch('http://localhost:3001/api/v1/auth/me', {
      headers: {
        Authorization: `Bearer ${accessToken}`,
      },
    });
    ```
  </Tab>
  <Tab value="Python">
    ```python
    import requests

    response = requests.get(
        'http://localhost:3001/api/v1/auth/me',
        headers={'Authorization': f'Bearer {access_token}'}
    )
    ```
  </Tab>
</Tabs>

## Session Lifecycle

<TypeTable
  type={{
    expires_at: {
      type: 'DateTime<Utc>',

      default: '30 days from creation',
    },
    last_seen_at: {
      type: 'DateTime<Utc>',
      description: 'Updated on every authenticated request. Useful for showing active sessions.',
    },
    revoked_at: {
      type: 'DateTime<Utc> | null',
      description: 'Set when the session is explicitly revoked via signout or admin action.',
    },
    device_name: {
      type: 'string | null',
      description: 'Provided by the client during signin, or inferred from the User-Agent header.',
    },
  }}
/>

A session is considered **invalid** if any of the following are true:

- `revoked_at` is set (session was explicitly revoked)
- `expires_at` is in the past (session expired)
- The user's `deleted_at` is set (user was soft-deleted)
- The user's `status` is not `active`

## Protected vs Public Endpoints

| Endpoint | Auth Required | Description |
|---|---|---|
| `POST /auth/signup` | ❌ No | Register a new account |
| `POST /auth/signin` | ❌ No | Sign in with credentials |
| `POST /auth/forgot-password` | ❌ No | Request a password reset OTP |
| `POST /auth/reset-password` | ❌ No | Reset password with OTP |
| `POST /auth/send-verification-otp` | ❌ No | Request email verification OTP |
| `POST /auth/verify-email` | ❌ No | Verify email with OTP |
| `GET /auth/oauth/{provider}/authorize` | ❌ No | Start OAuth flow |
| `GET /auth/oauth/{provider}/callback` | ❌ No | OAuth callback |
| `GET /auth/me` | ✅ Yes | Get current user |
| `POST /auth/signout` | ✅ Yes | Sign out current session |
| `POST /auth/signout-all` | ✅ Yes | Revoke all sessions |
| `POST /auth/change-password` | ✅ Yes | Change password |
| `GET /auth/sessions` | ✅ Yes | List user's sessions |
| `DELETE /auth/sessions/{id}` | ✅ Yes | Revoke a specific session |
| `GET /roles` | ✅ Yes + Permission | List roles |
| `POST /roles` | ✅ Yes + Permission | Create a role |
| `GET /permissions` | ✅ Yes + Permission | List permissions |
| `POST /permissions` | ✅ Yes + Permission | Create a permission |
| `POST /roles/{id}/permissions/{id}` | ✅ Yes + Permission | Assign permission to role |
| `DELETE /roles/{id}/permissions/{id}` | ✅ Yes + Permission | Remove permission from role |
| `POST /users/{id}/roles/{id}` | ✅ Yes + Permission | Assign role to user |
| `DELETE /users/{id}/roles/{id}` | ✅ Yes + Permission | Remove role from user |

<Callout type="warn" title="Authorization Layer">
  Endpoints under the **Authorization** section require the authenticated user to have specific permissions assigned via their roles. See the [Authorization docs](/docs/api-reference/authorization) for details.
</Callout>
