## Error Response Format

All API errors follow a consistent JSON structure:

```json title="Error Response"
{
  "error": {
    "code": "error_code",
    "message": "Human-readable error description"
  }
}
```

  type={{

## HTTP Status Codes

| Status Code | Error Code | Description |
|---|---|---|
| `400` | `bad_request` | The request body or parameters are invalid |
| `401` | `unauthorized` | Missing, invalid, or expired authentication token |
| `403` | `forbidden` | Authenticated but lacking required permissions |
| `404` | `not_found` | The requested resource does not exist |
| `409` | `conflict` | The request conflicts with existing data (e.g., duplicate email) |
| `429` | — | Rate limit exceeded (via Tower Governor) |
| `500` | `database_error` | Internal database error |
| `500` | `migration_error` | Database migration failure |
| `500` | `password_error` | bcrypt hashing/verification failure |
| `500` | `configuration_error` | Server configuration error |

## Common Error Scenarios

<Accordions type="single">
  <Accordion id="bad-request" title="400 Bad Request">
    Returned when the request payload fails validation.

    **Common causes:**
    - Invalid email format
    - Password shorter than 8 characters
    - Missing required fields
    - Invalid OTP code
    - OTP expired or already consumed
    - OTP max attempts exceeded
    - Unsupported OAuth provider
    - Invalid role name or permission key

    ```json
    {
      "error": {
        "code": "bad_request",
        "message": "bad request: password must be at least 8 characters"
      }
    }
    ```
  </Accordion>

  <Accordion id="unauthorized" title="401 Unauthorized">
    Returned when the request lacks valid authentication.

    **Common causes:**
    - Missing `Authorization` header
    - Malformed Bearer token (missing `Bearer ` prefix)
    - Token doesn't match any session
    - Session is expired (`expires_at` in the past)
    - Session was revoked (`revoked_at` is set)
    - User account is deleted or inactive
    - Wrong email/password during sign in

    ```json
    {
      "error": {
        "code": "unauthorized",
        "message": "unauthorized"
      }
    }
    ```
  </Accordion>

  <Accordion id="forbidden" title="403 Forbidden">
    Returned when the user is authenticated but lacks the required permission.

    **Common causes:**
    - Trying to list roles without `roles:read` permission
    - Trying to create permissions without `permissions:create` permission
    - Trying to manage user roles without `user_roles:manage` permission

    ```json
    {
      "error": {
        "code": "forbidden",
        "message": "forbidden"
      }
    }
    ```
  </Accordion>

  <Accordion id="conflict" title="409 Conflict">
    Returned when the operation conflicts with existing data.

    **Common causes:**
    - Registering with an email that's already taken

    ```json
    {
      "error": {
        "code": "conflict",
        "message": "conflict: email is already registered"
      }
    }
    ```
  </Accordion>
</Accordions>

## Handling Errors in Code

<Callout type="info" title="Best Practice">
  Always check the `error.code` field for programmatic error handling rather than parsing the `message` string, which may change between versions.
</Callout>

```js title="Error Handling Example"
const response = await fetch('/api/v1/auth/signin', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email, password }),
});

if (!response.ok) {
  const { error } = await response.json();

  switch (error.code) {
    case 'unauthorized':
      // Invalid credentials
      showError('Invalid email or password');
      break;
    case 'bad_request':
      // Validation failure
      showError(error.message);
      break;
    case 'conflict':
      // Duplicate resource
      showError('This email is already registered');
      break;
    default:
      showError('An unexpected error occurred');
  }
}
```
